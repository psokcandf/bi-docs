CDP identity stitching for Target profile updates

The Target exposure reporting payload does not change. MetaTagger still sends only `experience_name`, `experience_type`, and `event222`. The identity stitching layer is separate: it lets Treasure Data CDP update Adobe Target profiles for the same visitor later.

System Field Required behavior
MetaRouter `analytics.user().anonymousId()` Expose a stable anonymous visitor ID in the browser and forward it to Treasure Data CDP.
Adobe Target at.js `mbox3rdPartyId` Receive the same MetaRouter anonymous ID through `targetPageParamsAll()` before the first Target request.
Treasure Data CDP / Adobe Target API `thirdPartyId` Use the same ID as the first CSV column when updating Target profiles through the Bulk Profile Update API v2.
// Event or profile field MetaRouter should make available to CDP { "metarouter_anonymous_id": "2374724e-5b40-499a-999c-d2b2b66c0040", "target_third_party_id": "2374724e-5b40-499a-999c-d2b2b66c0040" }
# Adobe Target Bulk Profile Update API v2 CSV batch=thirdPartyId,profile.segment,profile.customer_tier,profile.cdp_audience 2374724e-5b40-499a-999c-d2b2b66c0040,high_intent,gold,target_test_audience 9e3e6c1f-2a7f-4a94-a8e6-8c9a6e17d111,returning_user,silver,target_test_audience
# Postman request example POST https://bi.tt.omtrdc.net/m2/bi/v2/profile/batchUpdate Content-Type: text/plain batch=thirdPartyId,profile.segment,profile.customer_tier,profile.cdp_audience 2374724e-5b40-499a-999c-d2b2b66c0040,high_intent,gold,target_test_audience

In the browser the Adobe Target field is named `mbox3rdPartyId`. In the bulk profile update file and API it is named `thirdPartyId`. Both must contain the same MetaRouter anonymous ID.

Final Adobe Analytics mapping

MetaTagger property Target source Meaning Expiration
`experience_type` `activity.name`, fallback to `activity.id` if name is missing Target Activity Name Visit or Activity
`experience_name` `experience.name` Target Experience Name Visit or Activity
`event222` Target decision/render counter Adobe Target Exposure Counter Counter event

This is the final agreed MetaTagger payload for Adobe Target: only `experience_name`, `experience_type`, and `event222` are sent by the bridge. MetaRouter can map these readable properties to the chosen Adobe Analytics eVars and event. Additional experience IDs, offer IDs, and offer names are intentionally not sent in this implementation.

Experience Name is read from `experience.name`, with fallback to `experience.id` if the name is missing. If `activity.name` is missing, `experience_type` falls back to `activity.id`.

Admin setup

  1. Open Adobe Analytics Admin.
  2. Select the report suite used by MetaRouter.
  3. Go to Conversion Variables.
  4. Create or rename `experience_type` as Target Activity Name.
  5. Create or rename `experience_name` as Target Experience Name.
  6. Create or reserve `event222` as the Adobe Target exposure counter.
  7. Use Text String type.
  8. Set allocation according to reporting needs; last touch is a practical default.
  9. Set expiration to Visit for visit-level attribution or Activity for longer conversion windows.

Workspace setup

  1. Create a Freeform Table.
  2. Use Target Activity Name as the first dimension.
  3. Break down by Target Experience Name.
  4. Add metrics such as Visits, Page Views, Orders, Leads, or custom conversion events.
  5. Create a segment where `event222` exists or `experience_type` exists to isolate exposed users.
  6. Compare exposed audiences using calculated metrics such as conversion rate or revenue per visit.

Map MetaRouter properties to Adobe variables

The bridge emits only the final readable MetaTagger properties. MetaRouter should map these properties into the Adobe Analytics hit without adding extra Target metadata.

Copy or adapt from: metarouter-loader.js and metarouter-target-bridge.js.

// MetaTagger / MetaRouter mapping example experience_type -> Target Activity Name -> Adobe Analytics eVar experience_name -> Target Experience Name -> Adobe Analytics eVar event222 -> 1 -> Adobe Analytics event222

Validate required response tokens

The bridge sends the MetaTagger event only when the required Activity Name and Experience Name values are present. Activity Name can fall back to Activity ID; Experience Name can fall back to Experience ID.

Enable `activity.name`, `activity.id`, `experience.name`, and `experience.id` response tokens in Adobe Target Administration.

// Required Target response token values experience_type = token["activity.name"] || token["activity.id"] experience_name = token["experience.name"] || token["experience.id"] event222 = 1

Build Workspace tables

Use activity name first, break it down by experience name when useful, then add exposure and conversion metrics.

MetaRouter event shape

The page bridge sends a normal analytics event through `metaTagger.track()`. The payload is intentionally limited to the final readable MetaTagger properties. The bridge also deduplicates identical Target tokens from the same Target response so the exposure counter is not inflated. The implementation lives in metarouter-target-bridge.js.

{
  "event": "Adobe Target Decision",
  "experience_name": "Experience B",
  "experience_type": "Country Allocation C&F",
  "event222": 1
}

Audience and impact analysis

Use Target exposure metadata to create audiences for each activity, then compare those audiences against the business events that matter. The agreed payload supports activity and experience-level exposure reporting using the agreed two eVars.

Audience Segment definition Example goal metrics Impact view
All Target-exposed users `event222` exists, or `experience_type` exists. Visits, Unique Visitors, Form Submit event, Download event. Use this as the denominator for exposure quality and total Target reach.
Specific activity exposed `experience_type` equals the tested activity name. Form Submit event, Lead event, Asset Download event. Conversion rate for users exposed to that activity.
Specific experience exposed `experience_name` equals the Target experience name, for example `Experience A`, `Experience B`, or the delivered experience ID. Same goal metrics as the activity audience. Compare performance by Target experience name.
Non-exposed comparison Exclude hits or visits where `event222` exists. Same goal metrics as exposed users. Use carefully as directional context, not as randomized-control proof.
Workspace calculated metrics Form submit rate = Form Submit event / Visitors with event222 Download rate = Download event / Visitors with event222 Activity conversion rate = Goal event / Visitors where experience_type equals the activity name Directional lift = (Exposed conversion rate - non-exposed conversion rate) / non-exposed conversion rate

Build one panel per activity, then use rows for `experience_type` and columns for event222, exposed visitors, goal events, and conversion rate. Keep the same attribution window across exposed and comparison audiences.

Validation checklist

  • Open the exact implementation files from this guide and adjust them to the client's final script paths, category IDs, and Analytics variables.
  • Response tokens are enabled in Adobe Target Administration.
  • The browser fires `adobe.target.event.REQUEST_SUCCEEDED` after Target returns a decision.
  • The bridge queues at least one `Adobe Target Decision` event.
  • MetaTagger receives the `metaTagger.track()` call and MetaRouter forwards the properties.
  • Adobe Analytics image/request payload contains `experience_name`, `experience_type`, and `event222` only for Target exposure reporting.
  • Workspace shows Target Activity Name, Target Experience Name, and event222 with the chosen success metrics.
