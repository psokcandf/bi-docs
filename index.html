Implementation guide

Evaluating Adobe Target when OneTrust controls consent and MetaRouter sends analytics

This guide explains how to run Adobe Target only after consent, prevent flicker, and forward the agreed Adobe Analytics payload through MetaRouter: `experience_name`, `experience_type`, and `event222`.

Default experience This top element is intentionally stable and easy to target. Adobe Target should modify this block when an activity qualifies.

How to use this package

This is example code. It can be modified freely for the client's implementation needs; it only demonstrates the recommended method, script responsibility, and loading order.

Why this setup is different

The client uses MetaRouter as a server-side routing layer for analytics. Because analytics collection is not the standard browser-side Adobe Analytics setup, the native A4T reporting path is not the right dependency for this proof.

The alternative is to collect Target decision metadata on the page and send it through MetaRouter as normal analytics fields.

What we collect

The bridge listens to the Target response event and reads only the agreed reporting fields: activity name and experience name. It sends `experience_name`, `experience_type`, and `event222` to MetaRouter. If `activity.name` is missing, `experience_type` falls back to `activity.id`; if `experience.name` is missing, `experience_name` falls back to `experience.id`.

Target response tokens must be enabled in Adobe Target Administration.

What developers implement

Developers add `targetPageParamsAll()`, the flicker handler, OneTrust, the consent wrapper, the MetaRouter identity resolver, the Target injection script, the Target analytics bridge, and the local or hosted `at.js` file.

Recommended sequence

This is the short version developers should remember. The files in this package are intentionally more defensive because they handle timing, retries, diagnostics, and consent changes. The core implementation principles are much simpler.

Set Target params before `at.js` runs

Adobe Target must know the property token and, for CDP profile stitching, the visitor ID before the first Target request is made.

// Principle: keep Target params available before loading at.js window.targetPageParamsAll = function() { return { at_property: "PROPERTY_TOKEN_FROM_ADOBE_TARGET", mbox3rdPartyId: "METAROUTER_ANONYMOUS_ID" }; };

Our implementation: adobe-target-injection.js builds this object and keeps the older page-specific params together with it.

Hide only the area Target can change

Use a short pre-hiding timeout to avoid flicker while Target evaluates. Prefer hiding the specific hero or module, not the whole page.

// Principle: hide the controlled area, then always reveal it again hide("#target-hero-message"); loadTarget(); show("#target-hero-message"); setTimeout(() => show("#target-hero-message"), 3000);

Our implementation: adobe-target-flicker-handler.js handles prior consent, timeout, and failure states.

Let OneTrust decide whether Target can load

Do not include `at.js` directly in the page. Load it only after the agreed OneTrust category is active.

// Principle: consent first, script injection second if (hasConsent(TargetProjectConfig.targetCategoryId)) { loadScript("./at.js"); }

Our implementation: target-consent-wrapper.js supports initial consent, consent changes, and script removal after revocation.

Use MetaRouter anonymous ID for Target profile stitching

The same ID has two names depending on where it is used: browser-side Target uses `mbox3rdPartyId`; Adobe Target profile update files use `thirdPartyId`.

// Principle: same value, different field names const id = analytics.user().anonymousId(); // Browser Target request params.mbox3rdPartyId = id; // CDP / Bulk Profile Update CSV thirdPartyId = id;

Our implementation: adobe-target-identity.js waits for `analytics.user().anonymousId()` and blocks Target loading briefly if the ID is not ready.

Send Target exposure separately from identity

Identity stitching is not the reporting event. Reporting still uses only the agreed Target exposure payload.

// Principle: readable exposure event for MetaRouter / Adobe Analytics metaTagger.track("Adobe Target Decision", { experience_name: "Experience B", experience_type: "Country Allocation C&F", event222: 1 });

Our implementation: metarouter-target-bridge.js reads Target response tokens and sends only these three reporting fields.

Configuration

  • Environment-specific values live in `.env` and are baked into `dist/target-config.js` by `./scripts/build.sh`.
  • Deploy the full `dist/` folder to a registered test or staging domain. OneTrust, Target, and MetaRouter do not fully work on localhost.
  • `at.js` is generated from `assets/at.js.template` during the build without the legacy inline `targetPageParams()` block.
  • MetaRouter receives Target exposure events through the dedicated `metaTagger.track()` method.

IT implementation checklist

Area Instruction Important note
Property token Fetch the property token from Adobe Target Administration > Properties, or request it from an Adobe Target administrator. The token must be passed through `targetPageParamsAll()` before `at.js` executes.
at.js library Download `at.js` from Adobe Target Administration > Implementation and host it on the client server or CDN. The hosted `at.js` should be a clean library file. Do not keep a hardcoded inline `targetPageParams()` block inside `at.js`.
Head sequence Load `target-config.js` and the flicker handler first, then `onetrust-loader.js`, the consent wrapper, MetaRouter loader, Target identity resolver, Target bridge, and Target injector. This order prevents direct Target loading before consent and reduces flicker on already-consented visits.
Identity stitching Use MetaRouter `analytics.user().anonymousId()` as the shared identifier. Send it to Adobe Target as `mbox3rdPartyId`; use the same value as `thirdPartyId` in Target profile update files. This is separate from the `metaTagger.track()` exposure payload and does not change `experience_name`, `experience_type`, or `event222`.
Consent category Set the Target OneTrust category in `.env` as `onetrust_target_category_id`, then mirror it in `AdobeTargetConsentConfig.categoryId` and `AdobeTargetFlickerConfig.categoryId`. The demo reads the category from generated `target-config.js`; the client category must match the approved OneTrust classification.
Flicker handling Configure the flicker handler selector to the smallest Target-controlled area possible, for example `#target-hero-message`. Use full-body hiding only when the whole first viewport can be personalized. The timeout fallback should stay short, typically 3000 ms.
MetaRouter payload Forward only `experience_name`, `experience_type`, and `event222` through `metaTagger.track()`. `experience_type` is Activity Name, `experience_name` is Experience Name, and `event222` is the Target exposure counter.
A4T limitation Do not rely on native A4T for this setup when MetaRouter owns analytics delivery and consent categories can differ. If Target is allowed but Analytics is not, the experience can render while native A4T reporting does not receive the Analytics hit.
Validation Check `adobe.target.VERSION`, the Target delivery request, response tokens, the `metaTagger.track()` call, and the final Adobe Analytics payload. Response tokens must be enabled in Adobe Target Administration for the bridge to populate `experience_type` and `experience_name`. `experience_type` can fall back from `activity.name` to `activity.id`; `experience_name` can fall back from `experience.name` to `experience.id`.

Target identity stitching for CDP profile updates

This identity flow is independent from the exposure reporting payload. The exposure payload remains `experience_name`, `experience_type`, and `event222`. The identity flow gives Treasure Data CDP a stable identifier it can use later to update the matching Adobe Target profile.

Place Field name Value
MetaRouter browser SDK `analytics.user().anonymousId()` The source identity for stitching.
Adobe Target at.js `mbox3rdPartyId` The same MetaRouter anonymous ID, sent before `at.js` makes the first Target request.
Adobe Target Bulk Profile Update API `thirdPartyId` The same ID in CDP-generated batch files.
// Browser-side Target params window.targetPageParamsAll = function() { return { at_property: "PROPERTY_TOKEN", mbox3rdPartyId: window.AdobeTargetIdentity.thirdPartyId, PageCheck: "target-demo" }; };

In DevTools Network, open the Adobe Target `delivery` request. The browser-side input name is `mbox3rdPartyId`, but at.js sends it to the Delivery API as `id.thirdPartyId`. The property token appears as `property.token`. The response should include response tokens such as `activity.name` and `experience.name` when the activity qualifies and response tokens are enabled in Adobe Target.
